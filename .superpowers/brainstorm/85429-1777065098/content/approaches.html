Spindle Demo Engine — Architectural Approaches

Three ways to build a full-featured demo engine. All share the same foundation; they differ in how much new structure gets added.

A

Thin Orchestration Layer

What it is: A new spindle demo CLI command with three subcommands (infer, stream, seed) that wire together existing Spindle features. A companion Fabric notebook template gets auto-generated per mode.

New code: ~500–800 LOC. No new modules — just orchestration scripts calling the existing engine, writers, and inference system.

Entry points: CLI for consultants, one pre-built notebook per mode for presenters.

Pros

  • Fastest to ship (1–2 sprints)
  • Minimal risk — no new architecture
  • Every mode works today with glue code

Cons

  • Hard-coded flows — no easy customization
  • Three separate notebooks, not unified
  • No progress visibility / observability
  • No client-specific scenario memory
B

Scenario Pack Architecture Recommended

What it is: Demo scenarios are YAML packs (Spindle already has a packs system — just undocumented). Each pack declares: what data to generate, which writers to target, which validation gates to run, and a cleanup spec. spindle demo run PACK_NAME executes the scenario. A notebook is auto-generated from the pack.

New code: ~1,200–1,800 LOC across: pack schema extensions (demo metadata), a DemoRunner that wraps PackRunner, a notebook generator, and a progress reporter.

Entry points: CLI (spindle demo run retail-inference --target myworkspace) + auto-generated Fabric notebook from the pack spec. One source of truth for both.

Pros

  • Builds on existing packs system (not reinventing)
  • Consultants can customize YAML per client
  • Notebook auto-gen means CLI + Fabric stay in sync
  • Extensible: add new scenarios without code changes
  • Can ship Phase 1 (inference pack) in 1 sprint

Cons

  • Packs system needs to be documented (was already undocumented)
  • More upfront design on pack schema
  • YAML learning curve for new consultants
C

Full Demo Engine Module

What it is: A new first-class sqllocks_spindle/demo/ module with: ScenarioCatalog (registered scenarios), DemoOrchestrator (multi-step execution with rollback), ProgressDashboard (Jupyter widget or Power BI streaming), and CostEstimator (CU impact before run). The cleanest product experience.

New code: ~3,500–5,000 LOC. New module, new abstractions, new widget/dashboard layer.

Entry points: CLI + Fabric notebook + optional local web dashboard.

Pros

  • Best end-user experience — unified, polished
  • Real-time progress + cost visibility
  • Full rollback on partial failure
  • Conference-quality wow factor

Cons

  • 3–5× more code than Option B
  • 2–3 sprints minimum before anything ships
  • Jupyter widget / PBI streaming is non-trivial
  • Most risk — new abstractions may conflict with existing engine

My recommendation: B — The packs system already exists and is production-code quality, just undocumented. Building the demo engine on top of it means we get extensibility and customization for free, ship Phase 1 (inference demo) in one sprint, and add streaming + seeding as additional packs without re-architecting. Option C is the right long-term vision but Option B delivers 80% of the value in 30% of the time.
