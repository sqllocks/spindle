What Could Be Better + What's Missing

10 gaps and design improvements found after digging into the actual code. Some are critical blockers; others are structural improvements.

Critical Blockers — Phase 1 won't land without these

1. No connection registry

The design has no spindle demo init — no way to store which Fabric workspace, Warehouse, Eventhouse to target. Every demo still requires manual credential entry, which killed ChrisHydeRules. This is the #1 friction point.

Fix: Add connections.py + spindle demo init command. Stores named profiles: spindle demo run retail-inference --connection mycompany.

2. ProgressDashboard is Phase 2 — wrong

Even for inference mode (Phase 1), the audience sees nothing happening. "Learning schema... analyzing 47 columns... generating 500K rows... comparing distributions" needs to be visible from day one. Without it Phase 1 is a black box.

Fix: Move output/dashboard.py (rich progress bar, step tracker) into Phase 1. ipywidgets version stays Phase 2.

3. powerbi.py is a lie

SemanticModelExporter already exists and generates .bim (TOM JSON) — not .pbix. Generating .pbix from scratch is not feasible (binary format, no public spec). The design file is misnamed and will confuse anyone reading the spec.

Fix: Rename to output/semantic_model.py. Wrap SemanticModelExporter + XMLA deploy. Be explicit: produces .bim, not .pbix.

4. No scenario parameterization

A consultant in a client meeting can't say "use my client's actual CSV" or "name the environment Contoso". Scenarios baked as Python files have no parameter surface.

Fix: Scenarios take a DemoParams dataclass: --input-file, --rows, --env-name, --connection. Pass through to all modes.

Design Improvements — make the architecture cleaner

5. scenarios/ splits by domain+mode combo

retail_inference.py + retail_streaming.py as separate files means every new domain needs 3 files. Mode logic leaks into scenario definitions.

Fix: scenarios/retail.py defines a RetailScenario that supports all modes. Mode is a parameter: spindle demo run retail --mode inference.

6. database_profiler.py is already built

The design implies live DB inference is future work. It's not — inference/database_profiler.py already connects via pyodbc + Entra ID token. The InferenceDemoMode can use live Fabric Warehouse connections today.

Fix: Update design to show that --input live-db is already supported. Expose it in the inference demo flow from day one.

7. Missing: spindle demo preflight

Before a conference talk: "can I write to all 4 targets? is auth working? can I hit the Eventhouse? does the workspace exist?" No pre-show health check means you find out on stage that something's broken.

Fix: Add spindle demo preflight --connection mycompany to Phase 1 CLI commands. Validates auth + targets before anything runs.

8. Missing: AdventureWorks scenario

The conference talk is literally titled "Retire AdventureWorks." There should be a built-in adventureworks scenario that generates AW-compatible schema and runs a head-to-head comparison. Without it the talk has no demo.

Fix: scenarios/adventureworks.py — generates DimCustomer, DimProduct, FactSalesOrder etc., then compares against real AW distributions. Should be Phase 1.

9. DemoResult needs an export format

After a demo runs, there's no shareable artifact showing what was created, where it lives, and how to clean it up. For workshop mode, attendees need this to take something home.

Fix: DemoManifest.export(format="md"|"html") generates a summary report. Add spindle demo report SESSION_ID command.

10. No dynamic scenario composition

To run retail + healthcare together without a pre-built scenario file, you're stuck. Dynamic composition would let consultants mix domains on the fly per client.

Fix: spindle demo run --domain retail,healthcare --mode seed --connection myco — ScenarioCatalog assembles the composite at runtime using existing presets system.
