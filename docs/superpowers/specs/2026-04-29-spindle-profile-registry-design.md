# Spindle Profile Registry — Design Spec

> **Date:** 2026-04-29
> **Author:** Jonathan Stewart
> **Status:** Approved (pending spec self-review)
> **Version target:** v2.12.0

---

## Overview

Spindle profiles today are scattered across `sqllocks_spindle/domains/<domain>/profiles/default.json` (built-in) and ad-hoc files generated by `LakehouseProfiler` / `DatabaseProfiler` / `DataProfiler` (user-generated). There is no consistent way to:

- Categorize profiles by source system (Salesforce, SAP, Fabric-prod, etc.)
- Maintain multiple profiles for the same logical table (dev/prod, quarterly snapshots)
- Search/filter across many profiles
- Bulk-import 150 tables from a real source in a single call
- Diff two profiles of the same table

This design adds a **lightweight profile registry** that sits alongside the existing inference APIs without breaking them.

---

## Architecture Principles

1. **Hybrid identity** — system + tags. System is a folder (rigid, identifies origin). Tags are arbitrary (flexible, identify lifecycle/owner/domain).
2. **Folder per table, file per profile** — `<system>/<table>/<profile_name>.json` allows multiple profiles per table without filename collision.
3. **Lightweight registry** — auto-generated `_index.json` with profile metadata for fast search. Index is regenerable; never source of truth.
4. **No remote/no GUI in v1** — local filesystem only. Defer S3/lakehouse-hosted registries and web UIs.
5. **Built-in domain profiles untouched** — `domains/<d>/profiles/default.json` keeps shipping defaults; registry is for user/customer profiles.

## Section 1 — Storage Layout

```
<project-root>/profiles/                      ← root, configurable
  _index.json                                 ← auto-generated, regenerable
  <system>/
    <table>/
      <profile_name>.json
```

**Identity:** `<system>.<table>.<profile_name>` (dot-separated). Example: `salesforce.customer.prod-2026Q2`.

**Root resolution:** `ProfileRegistry(root=...)` arg → `SPINDLE_PROFILES_DIR` env var → `./profiles` (cwd-relative).

**Reserved names:** `_index.json` (registry index), any name beginning with `_` is reserved.

---

## Section 2 — Profile File Schema

Each `<profile_name>.json` is a superset of today's `ExportedProfile`:

```json
{
  "name": "prod-2026Q2",
  "system": "salesforce",
  "table": "customer",
  "tags": ["prod", "customer-360", "pii"],
  "description": "SF customer table, prod sample 2026-04-15",
  "source": "Calibrated from 4.5M-row prod sample",

  "row_count_observed": 4500000,
  "schema_hash": "sha256:abc123...",
  "created_at": "2026-04-29T12:00:00Z",
  "updated_at": "2026-04-29T12:00:00Z",

  "distributions": { ... },
  "ratios": { ... },
  "table_profile": { ... },
  "metadata": { ... }
}
```

**Backward compatibility:** Files missing `system`/`table`/`tags` get those values inferred from the path on first read; values are persisted on next write. Built-in domain profiles read fine but are never auto-rewritten.

---

## Section 3 — _index.json (auto-generated)

```json
{
  "version": 1,
  "generated_at": "2026-04-29T12:00:00Z",
  "count": 47,
  "profiles": [
    {
      "id": "salesforce.customer.prod-2026Q2",
      "system": "salesforce",
      "table": "customer",
      "name": "prod-2026Q2",
      "tags": ["prod", "customer-360", "pii"],
      "row_count_observed": 4500000,
      "schema_hash": "sha256:abc...",
      "created_at": "2026-04-29T12:00:00Z",
      "path": "salesforce/customer/prod-2026Q2.json"
    }
  ]
}
```

Index is rebuilt on every `save()` / `delete()` / `import_from_*()`, and on demand via `reg.reindex()`. If `_index.json` is missing or corrupt, `ProfileRegistry` rebuilds it transparently from the file tree on first access.

## Section 4 — Python API

### Module: `sqllocks_spindle.profiles`

```python
from sqllocks_spindle.profiles import ProfileRegistry, RegistryProfile
```

### Class: `ProfileRegistry`

```python
class ProfileRegistry:
    def __init__(self, root: str | Path | None = None) -> None: ...

    # Discovery
    def list(
        self,
        system: str | None = None,
        table: str | None = None,
        tags: list[str] | None = None,  # AND-filter
    ) -> list[RegistryProfile]: ...
    def search(self, query: str) -> list[RegistryProfile]: ...   # fuzzy on table name
    def systems(self) -> list[str]: ...
    def tables(self, system: str | None = None) -> list[str]: ...

    # CRUD
    def load(self, profile_id: str) -> RegistryProfile: ...
    def save(self, profile: RegistryProfile) -> None: ...
    def delete(self, profile_id: str) -> None: ...
    def rename(self, profile_id: str, new_id: str) -> None: ...
    def tag(
        self,
        profile_id: str,
        add: list[str] | None = None,
        remove: list[str] | None = None,
    ) -> None: ...

    # Bulk import
    def import_from_lakehouse(
        self,
        workspace_id: str,
        lakehouse_id: str,
        system: str,
        tags: list[str] | None = None,
        profile_name: str = "latest",
        sample_rows: int | None = 100_000,
    ) -> list[str]: ...   # returns imported profile_ids
    def import_from_database(
        self,
        connection_string: str,
        system: str,
        tags: list[str] | None = None,
        profile_name: str = "latest",
        tables: list[str] | None = None,
    ) -> list[str]: ...
    def import_from_dataframe(
        self,
        df: pd.DataFrame,
        system: str,
        table: str,
        profile_name: str = "latest",
        tags: list[str] | None = None,
    ) -> str: ...   # returns profile_id

    # Compare
    def diff(self, id_a: str, id_b: str) -> dict: ...

    # Maintenance
    def reindex(self) -> None: ...
```

### Class: `RegistryProfile`

```python
@dataclass
class RegistryProfile:
    name: str
    system: str
    table: str
    tags: list[str]
    description: str
    source: str
    row_count_observed: int
    schema_hash: str
    created_at: str
    updated_at: str
    distributions: dict
    ratios: dict
    table_profile: TableProfile | None
    metadata: dict

    @property
    def id(self) -> str:           # "<system>.<table>.<name>"
        return f"{self.system}.{self.table}.{self.name}"

    def to_dataset_profile(self) -> DatasetProfile: ...
    def to_json(self) -> str: ...
    @classmethod
    def from_json(cls, data: str | dict) -> "RegistryProfile": ...
```

### Generation integration

```python
# Spindle.generate() gets a profile_id kwarg
result = Spindle().generate(
    profile_id="salesforce.customer.prod-2026Q2",
    scale="medium",
    seed=42,
)
# Internally: registry.load(profile_id).to_dataset_profile() → SchemaBuilder → generate
```

## Section 5 — CLI

```bash
spindle profile list                                       # all profiles
spindle profile list --system salesforce --tag prod        # filter
spindle profile show salesforce.customer.prod-2026Q2       # full JSON
spindle profile diff <id1> <id2>                           # field-by-field diff
spindle profile tag <id> --add prod --remove dev
spindle profile rename <old-id> <new-id>
spindle profile delete <id>
spindle profile reindex                                    # rebuild _index.json

# Bulk import — the 150-table workflow
spindle profile import-lakehouse <workspace-id> <lakehouse-id> \
    --system fabric-prod-lh --tag prod --tag fabric \
    [--profile-name latest] [--sample-rows 100000]

spindle profile import-database "<odbc-string>" \
    --system mssql-erp-prod --tag prod \
    [--tables customer,order,product]
```

All commands respect `SPINDLE_PROFILES_DIR`.

## Section 8 — Out of Scope (v1)

- **Versioning** — multiple snapshots per `<id>` with parent links. Defer; users name them `prod-2026Q1`, `prod-2026Q2` themselves.
- **Visual diff** — beyond text/JSON output. Defer.
- **Remote registry** — S3/lakehouse-hosted. Defer.
- **Web UI** — defer.
- **Profile sharing/sync** between machines. Defer (use git for now).
- **Built-in domain profile migration into registry** — domain profiles remain in their current location; registry is for user data.

## Section 9 — Implementation Order

1. `sqllocks_spindle/profiles/registry.py` — `ProfileRegistry` class + `_index.json` reader/writer
2. `sqllocks_spindle/profiles/profile.py` — `RegistryProfile` dataclass + JSON IO + `to_dataset_profile()`
3. `sqllocks_spindle/profiles/__init__.py` — public API
4. Tests: `tests/test_profile_registry.py` — CRUD, list/search, tag, save/load round-trip, index rebuild
5. Tests: `tests/test_profile_registry_bulk.py` — bulk import (mocked LakehouseProfiler / DatabaseProfiler)
6. CLI: `sqllocks_spindle/cli.py` — `spindle profile <subcommand>` group
7. Generation integration: `Spindle.generate(profile_id=...)` kwarg in `engine/generator.py`
8. Docs: `docs/guides/editing-profiles.md`, `docs/guides/profile-registry.md`
9. Demo notebooks: `notebooks/demos/07-10.ipynb`
10. Version bump v2.12.0 + changelog

## Section 10 — Testing Strategy

- **Unit tests:** `RegistryProfile` JSON round-trip, `ProfileRegistry` CRUD, filter logic, identity parsing.
- **Integration tests:** Mocked bulk import using `MockLakehouseProfiler` / `MockDatabaseProfiler` (extend Phase 5 mock infrastructure).
- **Regression tests:** Existing tests don't import `sqllocks_spindle.profiles`; new module must not change any existing behavior.
- **Live tests** (optional, `@pytest.mark.live`): Real `import_from_lakehouse(Fabric_Lakehouse_Demo)` round-trip.

Target: ~50 new tests, all green, runtime <30s for unit/integration, full regression unchanged.
